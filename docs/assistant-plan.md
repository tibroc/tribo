# Tribo AI Assistant — Plan

An assistant that helps the family keep track of chores, prioritize the day/
week, and get more out of reviews. Staged rollout: **Phase 1 — generated
briefs** (no chat, one scheduled LLM call), **Phase 2 — chat with tool use**
on the same backend plumbing. **Both phases are implemented** — see CLAUDE.md
"Current status" for what shipped.

UI exploration mockups (phone frames, Salvia design system, seeded Silva-family
data): see the "Tribo assistant — UI options" Claude artifact (options A–D).
**Decided:** option **B** (structured Home brief) is phase 1; option **D**
(chat bottom sheet) is phase 2. Option A (narrative digest) rejected; option C
(review coach's notes) not pursued.

## Guiding decisions

- **Provider-agnostic LLM backend.** One client speaking the OpenAI-compatible
  chat-completions API. Works with Anthropic (OpenAI-compat endpoint), Gemini
  (OpenAI-compat endpoint), Ollama, and vLLM. Config via env only:
  `ASSISTANT_BASE_URL`, `ASSISTANT_API_KEY` (optional for local backends),
  `ASSISTANT_MODEL`. Unset ⇒ the feature is disabled and completely hidden in
  the UI (same pattern as Radicale/OIDC gating).
- **Grounded, never generative-from-nothing.** Every brief/note is generated
  from real service data (the same queries behind `/api/briefing` and
  `/api/review`), passed as structured context. The model summarizes and
  prioritizes; it does not invent events.
- **Cheap by construction.** Phase 1 is ~1 call/day (+1/week for review
  notes), generated by the existing in-process scheduler and cached in SQLite.
  Manual "Refresh" is rate-limited.
- **Privacy stance documented.** A self-hosted family app sending data to a
  cloud LLM must say so: settings shows which backend is configured and what
  is sent (calendar titles, chore names, member first names — never PINs,
  OIDC subjects, or credentials). Local backends (Ollama/vLLM) keep everything
  in the house.

## Phase 1 — Structured brief (option B)

### Backend

- New package `internal/assistant`:
  - `client.go` — minimal OpenAI-compatible chat client (POST
    `{base}/chat/completions`, JSON mode). No SDK dependency; net/http only.
  - `context.go` — assembles the grounding payload from existing services:
    today's/this week's events (incl. `needs_guardian` conflicts + free
    guardians), chore instances with owners/streaks, work schedules, last
    week's completion stats, upcoming birthdays. Reuses `internal/{calendar,
    chores,todos,family}` — no duplicated queries.
  - `brief.go` — generates the **daily brief** (and weekly variant) as
    structured JSON: `{ priorities: [{title, why, memberId,
    choreInstanceId?/eventId?}], watchOut?, praise? }`. Structured output keeps
    rendering deterministic, makes i18n feasible, and suits small local models.
- Storage: migration `0007_assistant.sql` —
  `assistant_brief(id, kind 'day'|'week', period_start, content_json,
  model, created_at)`. Cache-of-record; regenerating replaces the row.
- Scheduler: nightly job generates the day brief (and the week brief on
  Mondays). Failures log and keep the previous brief — the card shows its
  generation time so staleness is visible.
- API:
  - `GET  /api/assistant/status` — configured? model? (drives UI gating)
  - `GET  /api/assistant/brief?kind=day|week` — cached brief
  - `POST /api/assistant/brief/refresh` — regenerate now (rate-limited)
- Language: briefs are generated server-side; the app's UI language is
  per-device, so the brief language is server config — `ASSISTANT_LANGUAGE`
  (default `en`), consistent with the rest of the env-only assistant config.
  Revisit (e.g. a family-level DB setting) if it grates.

### Frontend

- **Home brief card** (option B, structured): first card under the greeting
  hero, with a Today / This week segmented toggle. Priorities are actionable
  rows (checkbox completes the chore via existing endpoints; conflict rows
  deep-link to the event form / claim buttons), followed by an optional
  watch-out callout (danger tint) and praise callout (brand tint). Shows
  "Generated 6:00 AM · {model}" + Refresh. Hidden when
  `/api/assistant/status` says unconfigured.
- Settings (Family → App settings): assistant row showing backend/model
  status + the privacy note. Config itself stays env-only.

### Testing

- `internal/assistant` unit tests with a fake OpenAI-compatible server
  (httptest): context assembly, JSON parsing, degraded/unconfigured paths.
- Golden test for the grounding payload so prompt-relevant data changes are
  visible in review.

## Phase 2 — Chat (bottom sheet) — implemented

- **UI (option D):** a ✦ button (above the FAB) opens a bottom-sheet chat over
  any screen — no nav slot consumed. Streaming replies; a visible tool trace
  ("✦ add_todo ✓") under each assistant message; quick-reply chips.
- **Backend:** `POST /api/assistant/chat` (SSE). Tool-calling loop reusing the
  same tool surface as `internal/mcp` (get_today, get_briefing, add_event,
  add_todo, complete_todo, complete_chore, check_availability) — refactor tool
  definitions into a shared package so MCP and chat stay one implementation.
- **Guardrails:**
  - Write-tools require an in-chat confirm affordance before executing, or are
    executed and clearly reported with undo where the domain allows.
  - Child profiles: read + complete-own-chores only; no event/member mutation.
  - Conversation state: ephemeral per sheet-open session first (no history
    table); persist later only if it earns its keep.
- **Caveat:** OpenAI-compatible *tool calling* quality varies across local
  models; phase 2 should degrade to read-only Q&A when the configured model
  can't tool-call reliably.

## Later / out of scope for now

- Review "coach's notes" (mockup option C) — explored, decided against for now.
- PWA push for the morning brief (needs a push service).
- Voice input in the chat sheet.
- Assistant-initiated rebalancing proposals (weekly "here's a fairer rotation"
  diff to accept/reject).
